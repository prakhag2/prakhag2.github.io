Finding Real Vulnerabilities Without Guessing

May 2026 · Security · AI Agents

There's a version of AI-assisted security testing that sounds great in demos and falls apart the moment you point it at real code.

The pitch goes like this: take a powerful model, give it access to a codebase, tell it to find vulnerabilities. Maybe wrap it in an agent loop so it can read files and run commands. The model is smart enough to understand code, so it should be smart enough to find the bugs. Right?

In practice, what you get back is a list of twenty "findings" — half of them are OWASP top-ten descriptions copy-pasted onto your route handlers, a few are real patterns applied to wrong contexts, and maybe one or two are interesting but unverifiable. You can't tell which is which without reading every line of code yourself, which defeats the entire point.

The problem isn't the model's ability to reason about code. It's that finding real vulnerabilities requires knowing what actually executes, not just what could hypothetically go wrong. And you can't get there by reading source files and guessing.

What we're building toward

This post walks through an approach we built and ran against OWASP Juice Shop — a deliberately vulnerable Node.js application with over a hundred known security issues across authentication, injection, authorization, and cryptography.

The scanning pipeline. Static parsing feeds into dynamic tracing, which produces call trees that agents can reason over.

The core idea: instead of handing an agent a repository and hoping it comes back with realistic vulnerabilities, you build a pipeline that gives the agent something much more constrained and useful — a graph of what actually happened at runtime, annotated with formal security properties, so it can form and test specific hypotheses rather than generating plausible-sounding guesses.

We'll walk through why the naive approach fails, what static analysis actually gives you (and where it stops), how dynamic tracing fills the gap, and how to structure the whole thing so that an agent's work is verifiable at every step.

Why "just scan the code" doesn't work

Let's start with what most people try first.

You point a model at a codebase. Maybe you give it a system prompt that says "you are a security expert." Maybe you feed it files one by one, or let it browse with a tool. The model reads through route handlers, spots patterns that look like known vulnerability classes, and reports them.

Here's what that looks like against Juice Shop's login endpoint:

// routes/login.ts
models.sequelize.query(
  `SELECT * FROM Users WHERE email = '${req.body.email}'
   AND password = '${security.hash(req.body.password)}'`,
  { model: UserModel, plain: true }
)

An agent reading this file will correctly flag SQL injection. The string interpolation of req.body.email into a raw query is textbook. This is the easy case — the vulnerability is visible in a single expression, no context needed.

But most real vulnerabilities aren't like this. They live in the interaction between components. Consider this pattern from the same codebase:

// routes/basket.ts
module.exports = function retrieveBasket () {
  return (req, res, next) => {
    const id = req.params.id
    BasketModel.findOne({ where: { id }, include: [ProductModel] })
      .then(basket => {
        if (basket?.Products?.length > 0) {
          for (let i = 0; i < basket.Products.length; i++) {
            basket.Products[i].name = req.__(basket.Products[i].name)
          }
        }
        res.json(utils.queryResultToJson(basket))
      })
    }
}

Is this vulnerable? Reading just this file, you might say "no — it queries by ID from URL params, but Sequelize parameterizes findOne by default." And you'd be partially right. The query itself is safe. But the security question is: does anything verify that the basket belongs to the requesting user?

That answer lives somewhere else — in a middleware, in how the route is registered, in what the JWT token contains and whether it's checked. An agent reading files in isolation can't answer this reliably. It would have to guess whether authorization exists upstream, and that guess is where most false positives come from.

Static analysis: what it gives you, honestly

Before going further, let's be clear: static analysis is valuable and complementary to everything discussed here. Tools like Semgrep, CodeQL, and Bandit catch real bugs — hardcoded secrets, unsafe deserialization, SQL injection via string formatting, use of deprecated crypto. They work by matching patterns against the AST without running the code.

Static analysis findings for a Juice Shop route. Pattern-matching catches the obvious — string interpolation in queries, hardcoded secrets, weak hashing.

For the class of issues where the vulnerability is self-contained in one expression or one function body, static analysis is the right tool. It's fast, deterministic, and scales to massive codebases without needing the application to run.

But it has a fundamental limitation: it reasons about what code could do, not what it does do. Consider:

  • A route handler that looks unprotected might have auth middleware registered at the router level, invisible to per-file analysis.
  • A function that takes unsanitized input might never be called with attacker-controlled data in practice.
  • A weak crypto function might exist but only run in test configuration.
  • A taint path that looks exploitable might be unreachable because of runtime type coercion or early-return conditions.

Static analysis can't resolve these without modeling the entire application's runtime behavior — at which point you're effectively building an interpreter, not a pattern matcher.

This is where dynamic tracing enters. Not as a replacement for static analysis, but as the piece that answers "does this actually happen?"

Capturing what actually executes

The approach starts with instrumenting the application at the module loading level. For a Node.js application like Juice Shop, that means hooking into the module compiler itself:

// tracer-hook.mjs — activated via: node --import ./tracer-hook.mjs
import { register } from 'node:module';
register('./tracer-loader.mjs', import.meta.url);

globalThis.__trace_enter = function(funcName, file, line, args) {
  const parentId = callStack[callStack.length - 1] || null;
  const id = nextId++;
  traces.push({
    id, parent: parentId, depth: callStack.length,
    func: funcName, file, line,
    input: serialize(args), output: null
  });
  callStack.push(id);
  return id;
};

globalThis.__trace_exit = function(traceId, returnValue) {
  const entry = traces.find(t => t.id === traceId);
  if (entry) entry.output = serialize(returnValue);
  // pop stack...
};

When Juice Shop loads, every function gets instrumented via Babel — transformed in memory, not on disk — so that entering and exiting any function records its arguments, return values, parent caller, and position in the call stack. No source files are modified. The application runs normally, just observed.

Then you exercise the application. In our case, we run Juice Shop's own API test suite — over a hundred test cases covering authentication, basket manipulation, user registration, file uploads, payment workflows, and more. Each test produces a complete execution trace.

A runtime trace for the 2FA verification flow. Every function call, its inputs, outputs, and position in the call stack — captured from an actual test execution.

Here's what one entry looks like in the raw trace — the application's 2FA verification endpoint, exercised by a test that sends a forged token:

{
  "id": 1,
  "parent": null,
  "depth": 0,
  "func": "_ret0",
  "file": "routes/metrics.ts",
  "line": 2,
  "input": {
    "req": {
      "__type": "Request",
      "method": "POST",
      "url": "/rest/2fa/verify",
      "body": { "tmpToken": "unsigned.jwt.here", "totp": "123456" },
      "headers": { "content-type": "application/json" }
    }
  },
  "output": "<void>"
}

This isn't speculation about what might happen. This is what did happen. The function was called. These exact values flowed through. And because the trace captures the parent-child relationship between calls, you can reconstruct the full path from HTTP request to database write to response.

Why this works especially well for new code

Here's a practical advantage that isn't immediately obvious: dynamic tracing is scoped to what you exercise. You don't need to analyze the entire codebase. If you just merged a new feature — say, a coupon redemption endpoint — you write or run the tests for that feature, capture the trace, and analyze only the paths that executed.

Compare this to running static analysis across the entire project after every PR. On a codebase the size of Juice Shop (hundreds of source files, thousands of routes and models), a full static scan produces noise from unchanged code that drowns out the signal from your new changes. Dynamic tracing naturally focuses on the delta — you trace the new functionality, and what you get back is the security-relevant behavior of exactly that code.

From traces to call trees to a graph you can query

A flat list of function calls is useful but hard to reason over. The next step transforms traces into call trees — hierarchical structures showing which functions called which, what data passed between them, and where in the application each call lives.

The call tree for a login flow, loaded into Neo4j. Red nodes read from user input. Blue nodes validate. Green nodes write to session storage. Edges show actual execution order.

These trees get loaded into Neo4j — a graph database where you can query relationships. "Show me every path from a node that reads HTTP input to a node that writes to the database" becomes a Cypher query, not a manual code review.

But here's the thing — a raw call tree still doesn't tell you what's security-relevant. A function called findOne might be reading user credentials or fetching a product image. The call tree shape is identical. You need to know what each node means in security terms.

That's where the ontology comes in.

Giving code formal security meaning

This is the piece that makes the whole approach deterministic rather than probabilistic. Instead of asking an agent "does this look vulnerable?", you classify every function in the execution graph using a formal ontology — a structured vocabulary of security roles.

The ontology defines what functions are based on what they do:

class InputReader(FunctionNode):
    """Reads from user-controlled source (HTTP headers, body, query)."""

class SessionWriter(FunctionNode):
    """Writes to session storage or authentication state."""

class Validator(FunctionNode):
    """Checks preconditions. Raises or returns error on failure."""

class CryptoOp(FunctionNode):
    """Performs cryptographic operation (hashing, signing, verifying)."""

class TaintSource(FunctionNode):
    """Origin of attacker-controlled data."""

Classification happens automatically, based on the trace data itself. A function that receives an Express Request object and reads req.body gets classified as InputReader. A function that calls jwt.verify() gets classified as CryptoOp. A function that calls session.save() gets classified as SessionWriter. No hardcoded function names — the patterns match on observed behavior.

A Cypher query finding paths from taint sources to session writers with no validator in between. Each result is a specific, testable hypothesis — not a guess.

Once the graph is annotated, you can ask structural questions that have deterministic answers:

// "Find every path from attacker-controlled input to a session write
//  that has NO validator or crypto check in between"

MATCH path = (src:TaintSource)-[:CALLS*1..6]->(sink:SessionWriter)
WHERE src.test = 'test_login'
  AND NONE(n IN nodes(path)[1..-1]
       WHERE n:Validator OR n:CryptoOp)
RETURN path

If this query returns results, you have a concrete path to investigate — not a heuristic match, not a model's best guess, but a structural fact about the application's runtime behavior. The taint source is specific (this function, this file, this line). The sink is specific. And the absence of defense is verifiable.

Why you can't just point an agent at a repo

Now let's come back to why the naive approach — "here's the codebase, find vulnerabilities" — produces unreliable results.

When you give an agent a large repository and ask it to find security issues, it faces an impossible search problem. Juice Shop has hundreds of source files. The agent needs to:

  1. Understand the application's routing structure
  2. Trace data flow across middleware, route handlers, models, and utilities
  3. Determine which inputs are attacker-controlled
  4. Figure out what defenses exist and whether they're applied consistently
  5. Distinguish between intentional design choices and missing checks

Doing all of that by reading files is like trying to understand a city's traffic patterns by looking at a map. The map tells you where roads exist. It doesn't tell you which ones are actually used, where the bottlenecks form, or which intersections have broken signals.

Here's what actually happens when you try. We ran an experiment: same model, same codebase, two approaches.

Approach A — agent with file access only:

Agent reads routes/login.ts
→ "Potential SQL injection in email parameter"      ✓ (real)
→ "No rate limiting on login attempts"              ✓ (real, but low-value)
→ "Password comparison may be timing-vulnerable"    ✗ (wrong — uses hash comparison)

Agent reads routes/basket.ts
→ "Missing authorization check on basket access"    ? (maybe — can't verify without knowing middleware)
→ "No input validation on basket ID parameter"      ✗ (Sequelize parameterizes)

Agent reads lib/insecurity.ts
→ "JWT signed with weak secret"                    ? (depends on config)
→ "authorize() function doesn't verify token exp"  ? (need to see how it's called)

Three definitive answers out of seven findings. The rest require context the agent doesn't have.

Approach B — agent with annotated call graph:

Planner queries Neo4j:
  "Paths from TaintSource to DBWriter with no Validator"
→ Returns 3 specific paths with file locations and line numbers

Attacker reads source along each path, generates hypothesis:
  "In test 'basket_retrieval': req.params.id flows directly to
   BasketModel.findOne() with no ownership check. The JWT middleware
   sets req.user but nothing compares req.user.id to basket.UserId."

Verifier runs exploit test:
  POST /api/Baskets/2  (with user1's token, accessing user2's basket)
→ 200 OK, returns basket contents
→ CONFIRMED: Broken Object-Level Authorization (BOLA)

Same codebase. The difference is what the agent started with — not a haystack of files to read, but a specific structural finding to verify.

The three-agent pipeline

This is where it all comes together. Instead of one agent trying to do everything, the work splits across three agents with different capabilities and constraints.

Each agent sees only what it needs. The Planner works from graph topology. The Attacker reads source along identified paths. The Verifier runs actual tests.

The Planner

The Planner works only from graph topology. It cannot read source code. It queries the Neo4j graph across all traced test cases and identifies every attack surface — every path from a taint source to a sensitive sink, every validator that gates a security decision, every place where a defense exists in one flow but is missing from another.

Its output is a prioritized target list. Each target is a specific hypothesis:

{
  "id": "T1-RemoteUser-Header-to-Login-SessionWriter",
  "priority": "CRITICAL",
  "path": [
    "get_username (TaintSource)",
    "process_request (Tainted)",
    "authenticate (Tainted)",
    "login (SessionWriter)"
  ],
  "property": "Authentication must verify identity through a
               cryptographic secret",
  "attack_angle": "If REMOTE_USER header can be injected,
                   attacker gets full session as any user.
                   No password check exists in this path."
}

Notice what this is: not "there might be a vulnerability somewhere in auth." It's a specific chain of functions, a specific security property that should hold, and a specific way it might be violated. The Planner can generate this because the graph gives it structural facts to reason over, not code to interpret.

The Attacker

The Attacker receives one target at a time. Unlike the Planner, it can read source code — but only along the path identified in the target. It reads each function on the chain, understands the exact conditions under which the defense holds or fails, and determines reachability: can an external attacker, controlling only HTTP requests, actually trigger this path?

This is where most naive agent findings die. The Attacker has strict reachability rules:

  • Attacker CAN control: HTTP headers, request body, cookies, URL params, timing
  • Attacker CANNOT control: server config, environment variables, installed middleware, database contents directly

If a finding requires the attacker to know a secret key or modify server configuration, it's classified as a hardening recommendation, not a vulnerability. This single constraint eliminates the majority of false positives that plague naive scanning.

When a path is reachable, the Attacker writes a complete test — real HTTP requests through the application's test client — that attempts to violate the security property.

The Verifier

The Verifier is the skeptic. It receives the Attacker's hypothesis and test code and independently evaluates whether the finding is real. It runs the test. If the test fails (meaning the security property was violated), it checks whether the failure actually demonstrates the claimed vulnerability or is just a test setup bug.

This adversarial structure matters. The Attacker is incentivized to find issues. The Verifier is incentivized to disprove them. A finding that survives both is substantially more likely to be real than one produced by a single agent acting as both prosecutor and judge.

Results from a full pipeline run. Twenty targets identified from the graph, each with a specific verdict and evidence. No ambiguous "maybe" findings.

Making it deterministic

Let's step back and look at what's structurally different about this approach compared to "agent reads code, reports issues."

The ontology removes interpretation. When the system says a function is a TaintSource, that's not a model's opinion. It's a classification based on observable facts — the function received an HTTP request object and read from its body. When it says a path has no Validator, that's a structural absence in the graph, not a judgment call about whether some code "looks like" validation.

The graph query is reproducible. Run the same Cypher query against the same graph and you get the same paths every time. The agent's role isn't to discover these paths through creative exploration — it's to verify whether they represent real exploits. The discovery step is deterministic. Only the verification requires reasoning.

The hypothesis is falsifiable. Every target has a specific security property: "authentication must verify identity through a cryptographic secret." The test either violates that property or it doesn't. There's no subjective severity rating that depends on how worried the agent feels. Property holds = not vulnerable. Property violated = vulnerable. Binary.

The evidence is executable. A confirmed finding comes with a test that anyone can run. Not a description of a theoretical attack, not a paragraph explaining why something "could be" dangerous — a script that demonstrates the behavior. If you disagree with the finding, run the test and see what happens.

How the pieces compose

Here's the full flow, end to end:

1. Instrument the application (Babel hook, no code changes)
2. Run tests → capture execution traces (per test case)
3. Build call trees from traces (parent-child relationships)
4. Classify nodes via ontology (TaintSource, Validator, SessionWriter...)
5. Export to Neo4j (nodes = functions, edges = calls + data flow)
6. Planner queries graph → prioritized target list
7. Attacker reads source along each path → reachability + test code
8. Verifier runs tests → confirmed / false positive / hardening

Steps 1-5 are fully automated and deterministic. Given the same application code and same test suite, they produce the same graph every time. Steps 6-8 use AI agents, but constrained to specific tasks with verifiable outputs.

The package graph shows module-level relationships. Combined with the call tree, it reveals which security boundaries the application actually enforces vs. which ones only exist conceptually.

Fitting into a development workflow

One more thing worth making explicit: this approach works best when it's scoped, not when it's global.

Running a full static analysis sweep across an entire project after every change produces diminishing returns. You get the same fifty findings every time, most of which are either already known or intentionally accepted. The signal-to-noise ratio degrades with scale.

Dynamic tracing inverts this. You trace what you changed. A new endpoint, a refactored auth flow, a feature branch that touches payment logic — you run the tests for that specific thing, capture the trace, build the graph for those paths, and analyze only the new attack surface.

This makes it practical for PR review. A developer adds a new API endpoint:

1. Tests for the new endpoint run (they exist because you wrote them)
2. Tracer captures execution → new trace file
3. Ontology classifies the new functions
4. Graph query checks: does this new code path have the same
   defenses as existing similar paths?
5. If not → specific, actionable finding attached to the PR

The question isn't "is there anything wrong with the entire codebase?" It's "does this new code follow the same security contract as the code around it?" That's answerable. And it's answerable without reading every file in the repository.

What this doesn't replace

To be clear about boundaries:

Static analysis still catches what it catches. Hardcoded secrets, string-interpolated queries, use of banned functions, dependency vulnerabilities — these are pattern-match problems and static tools handle them well. Run both. They're complementary, not competing.

Manual review still matters for design flaws. If your threat model is wrong — if you're trusting a header you shouldn't trust in the first place — no amount of tracing will tell you the design is broken. Tracing can tell you the header reaches the session writer without validation. It can't tell you whether trusting that header was ever a good idea.

Coverage depends on your tests. Dynamic tracing only captures what executes. If you don't have a test that exercises the vulnerable path, it won't appear in the trace. This is a feature (focus) and a limitation (blind spots) at the same time. The mitigation is straightforward: better test coverage gives you better security coverage. Which is true regardless of what analysis you run.

The underlying principle is simple: don't ask an AI to guess what might be vulnerable. Give it facts about what actually executes, give those facts formal meaning, and ask it to verify whether specific security properties hold. Structure the problem so that the AI's job is reasoning over evidence, not generating hypotheses from thin air.

The gap between "agent that scans code" and "agent that finds real vulnerabilities" isn't intelligence. It's the quality of the questions you let it ask.
