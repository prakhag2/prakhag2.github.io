Route the thread, not the message

June 2026 · LLM Systems

Most chat apps make two quiet decisions on every turn, and both are usually wrong: they send the message to one fixed model, and they hand that model the entire conversation so far.

The first is wasteful. A real conversation is a mix of "what's the capital of Norway" and "debug this race condition in my async code." Pin everything to your biggest model and you pay frontier prices to answer trivia; pin it to a small one and the hard turns come back wrong. One model size does not fit a conversation.

The second is worse, and less obvious. Conversations don't stay on one topic — people interleave. A user plans a trip, pauses to debug a script, comes back to the trip, asks about a recipe, returns to the script. Concatenate that whole transcript and feed it to the model on every turn, and you're asking it to answer a question about Tokyo hotels while staring at forty lines of Python traceback. The context isn't just expensive. It's the wrong context.

The transcript is the wrong unit

The instinct is to treat a conversation as a single linear thing — one list of messages, growing forever. But that's not how the underlying conversation actually works. What's really happening is several parallel lines of intent, woven together in time.

Think of it the way Slack does. A busy channel isn't one conversation; it's many threads sharing a timeline. A message belongs to a thread, not to the channel's running tail. The reply you want depends on which thread you're in, not on whatever happened to be said most recently.

A conversation is a set of open threads sharing a timeline — not one transcript with a single growing tail.

If you take that seriously, two things fall out for free. Each thread can be routed to the model that suits its topic — the coding thread to a strong coding model, the trivia thread to a cheap one. And each thread carries only its own history, so the model answering about Tokyo never sees the Python traceback. Routing and context isolation stop being two problems. They're the same decision: which thread does this message belong to?

So the whole system reduces to one question asked on every turn — which thread? — and everything downstream follows from the answer.

A conversation in motion

1"My Python script throws a KeyError."new threadt1Claude Sonnet 4 (tech)
2"How many days in Tokyo vs Kyoto?"resume t0t0Qwen3-32B

Turn 2 lands on the travel thread even though the message right before it was about Python. Turn 41 finds the same travel thread forty turns later. No topic was announced — nobody typed "anyway, back to Japan." The decision was made from the content alone. That recovery, after arbitrary gaps and with no cues, is the part worth getting right.

Deciding which thread — resume vs. new

There is no "current thread" and no clever heuristic about recency. On every turn, one small, cheap model is asked to place the message against the full set of open threads. It is shown exactly three things:

  • Each open thread, as a short summary — its objective plus a running scope: an accumulating list of every sub-topic that thread has touched. Crucially, the model does not see each thread's full message history. It sees the scope, which is the thread's compact, matchable identity.
  • The last few full turns of the whole chat — both the user message and the assistant reply, the last three by default. These exist only to resolve references like "yes," "the second one," or "and how much would that cost?" They're conversation-wide, not per-thread.
  • The new message — just the question.

It replies with one of two things:

{"thread": "t0"}                              → resume an existing thread
{"thread": "NEW", "objective": "plan trip"}   → open a new one

And then a guardrail that matters more than it looks: a returned id counts as a resume only if it's actually an open thread. A literal "NEW", a malformed reply, a hallucinated id that doesn't exist, an outright error from the model — all of them fall through to opening a new thread. The failure mode is biased on purpose. The model can never attach your message to the wrong thread by inventing an id; the worst it can do is start a fresh one, which the next turn can quietly correct.

Within that structure, the model is told to prefer resuming, and the rules it follows are the whole game:

  • Attach if the message fits a thread's objective or any sub-topic in its scope. A trip thread already covers transport, lodging, food, budget, etiquette; a budget question is part of it, not a new thread.
  • A return after a gap belongs to the thread it returns to. Scan all open threads, old ones included — don't open new just because the match is old or its scope has grown long. This is what makes turn 41 work.
  • Subject wins over recency. The thread touched most recently has no special status. Turn 2 resumes the travel thread despite the Python turn sitting immediately before it.
  • Beware look-alikes. "Build a transformer," "explain Bayes," "deploy on AWS," and "debug this pandas script" all share words — code, model, python, docker — yet they're four different threads. Match on the specific objective, not the shared vocabulary. This is the rule that stops everything from collapsing into one giant "tech" thread.
  • But concept and its implementation are one goal. "What is a transformer?" then "code it" then "what about positional encodings?" is a single thread, not three.

Notice what's not here: there is no merge step. Two threads are never fused into one. "Resume" simply means attaching to a thread that already exists. Because the decision is remade from scratch every turn, there's no state to get stuck in — a single misplaced turn doesn't corrupt anything; the following turn is judged fresh.

The model is chosen once, then inherited

When — and only when — a new thread is born, a second cheap classifier reads its objective and assigns a capability: a routing bucket like tech_coding, travel_planning, or a catch-all general_knowledge. That bucket maps to the model that answers the thread:

CapabilityAnswering model
tech_codingClaude Sonnet 4
travel_planningQwen3-32B
general_knowledge (default)Nova Lite

The classification happens once, at thread creation. Every subsequent turn that resumes the thread inherits its capability — it is never re-classified. That's deliberate: if you re-ran the classifier per message, a terse follow-up like "and the second one?" could get mislabeled and bounce the same thread between models mid-conversation. Pinning the model to the thread, not the message, makes routing stable. Turns 0, 2, and 41 all answer from the same travel model without anyone re-deciding.

The small steps — matching, capability classification, summarizing — all run on the cheapest model in the lineup. You only spend the strong model on the thing the user actually reads: the answer.

Each model sees only its own thread

Once the thread is chosen, the answer is generated from that thread's history and nothing else. When turn 2 is answered, the travel model receives only the turns belonging to t0, plus that thread's accumulated scope:

system:    You are a helpful assistant handling a travel_planning request.
           Context so far in this thread: <t0's accumulated scope>
user:      Planning a two-week Japan trip in April.     # t0, turn 0
assistant: <the trip reply>
user:      How many days in Tokyo vs Kyoto?             # the current turn

The Python turn from t1 is simply not in the prompt. It isn't summarized, truncated, or down-weighted — it's absent, because it belongs to a different thread. Conversely, when turn 3 resumes the coding thread, its model sees the KeyError turns and never hears about Japan. Each model gets a clean, on-topic context window. That isolation is the entire point: it's what makes the answers better and the prompts smaller at the same time.

Why the scope accumulates

After every answer, a summarizer folds the new question-and-reply into the thread's scope — and it's instructed never to drop earlier items. The scope grows into a complete, compact inventory of what the thread has covered: "Japan trip: April, two weeks, Tokyo vs Kyoto split, rail pass, daily budget…"

This is the quiet workhorse of the whole design. The scope is exactly what the matcher compares new messages against, so a rich, non-lossy scope is what lets turn 41 find its way home. A rolling summary that drifted toward the latest sub-topic would slowly forget the trip's early details, and the budget question forty turns later would look like a stranger. Accumulate instead of overwrite, and old aspects stay matchable for as long as the conversation lives.

What this doesn't solve

The threading decision is a model call, so it isn't free and it isn't perfect. A wrong attach sends a turn to a slightly-off context or a non-ideal model — a quality blip, not an outage, and one the next turn can correct. The design leans on that self-correction rather than pretending the classifier is always right.

It's also not a throughput trick. Routing easy turns to cheaper models saves money and frees your strong model for the turns that need it, but the ceiling is still the sum of what your models can actually serve. This makes spend efficient and context clean; it doesn't conjure capacity.

And the taxonomy is yours to write. The buckets — tech, travel, general, or whatever your product is about — and the descriptions the classifier reads are a config file, not a law of nature. Get them wrong and you'll route well-formed threads to the wrong model. The mechanism is domain-agnostic; the judgment about your domain isn't.

The shift is small to state and changes everything downstream: stop treating a conversation as one transcript bound to one model. Treat it as a set of threads, each with its own objective, its own model, and its own private context. Then every turn is just one cheap question — which thread does this belong to? — and routing, cost, and context all fall out of the answer.
